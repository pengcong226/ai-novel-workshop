# AI小说工坊 - 核心技术难点攻关总结

## 核心问题：100万字长篇小说的上下文管理

### 问题分析

**GPT-4/Claude上下文窗口：约128K-200K tokens**

100万字小说的上下文需求：
- 完整内容：约700,000 tokens（远超窗口限制）
- 最近10章：约35,000 tokens（可接受）
- 最近3章完整+10章摘要+检索：约20,000 tokens（最优方案）

### 解决方案：分层记忆系统（V5 最终落地）

```
┌─────────────────────────────────────────────────────────────┐
│                     Token预算分配                            │
├─────────────────────────────────────────────────────────────┤
│  系统提示 + 【铁律区】(死亡/封印状态)   2,000 tokens (最高优先)     │
│  实体与状态事件信息                 1,800 tokens (精确匹配)     │
│  活跃人物信息 + 近期经历              1,500 tokens (动态抽取)     │
│  Entity & StateEvent 状态记忆系统   1,500 tokens (动态更新)     │
│  长期记忆 (OP-RAG 向量保序检索)     2,000 tokens (时序补充)     │
│  短期记忆 (最近3章完整上下文)       6,000 tokens (固定截断)     │
│  当前章节细纲推演                    500 tokens (固定)          │
├─────────────────────────────────────────────────────────────┤
│  总计                         约15,300 tokens               │
│  剩余给生成                   约8,000 tokens (充足留白)     │
└─────────────────────────────────────────────────────────────┘
```

## 关键技术点

### 1. 向量检索实现 (OP-RAG 算法落地)

**选型：** Tauri 进程内置 `instant-distance` 库（原生 HNSW Rust 实现，脱离浏览器与 Chroma 限制）。

**嵌入模型：**
```
本地优先：@xenova/transformers (bge-small-zh-v1.5)
- 维度：512
- 部署：无需外部服务器，完全离线
- 中文支持：优秀
```

**OP-RAG 保序逻辑：**
传统 RAG 按照 Score 召回，导致前文碎片时间线错乱。
现已通过 `vector-service.ts` 的排序拦截，强行按 `.metadata.chapterIndex` 升序拼装。大模型获得的是按先后发生顺序排列的辅助文档。

### 2. 记忆更新与切片流程 (Zero-Touch Extract)

**选型：** 结构化 JSON Schema / Tool Calling（原正则抓取存在死锁问题）。

```
长达数万字的章节产出
      ↓
(V5新增) 语义边界拦截器 (判断是否有死亡、破境等"高影响词汇")
      ├─ 只有日常聊天 ─► 拦截丢弃，跳过提取节省开销
      └─ 有突破点 ───► 继续
      ↓
交给本地低价小模型 (如Ollama Qwen-7B / GLM) 进行 `state_extraction`
      ↓
将新人物/世界观直接更新入 SQLite 的 `ProjectStore` 和 Entity/StateEvent 系统
```

### 3. 一致性检查机制 (ConStory-Bench 十九项矩阵)

这是整个架构中最具防御力的环节。

**检查时机：** `antiRetconValidator.ts` (基于 19 项设定的生成后检查)
如果新生成的章节通过 JSON 解析出来包含矛盾（比如死人复活，技能降级）。

**处理流 (Consult Planner)：**
1. 拦截该次生成，拒绝落笔。
2. 将矛盾详情扔给**规划师 (Planner) Agent**。
3. Planner 分析是否是前文明明有“吃神药复活”的剧情而校验器不知道，如果判定合理，修改实体或状态事件。
4. 如果判定是写手 (Writer) AI 真的发疯吃书了，则附带“严厉的警告说明”要求其重写。

### 4. 混合存储架构 (Desktop+Web)

**选型：** 
由于浏览器端 `IndexedDB` 在写入 50MB (百万字) 以上小说单 JSON 文件时会面临极其严重的序列化延迟，甚至导致 Vue 页面卡死。

系统利用 `Tauri IPC` 隧道构建了双栖架构：
- 如果在浏览器：退化到 `IndexedDB` 凑合跑。
- 如果在桌面端：无缝切换为使用 `lib.rs` 的 SQLite 连接池，所有的实体与状态事件、项目文件拆分为不同表。极大地加速了响应。

### 5. 命运织布机与伏笔前瞻 (Plot Loom & Anchor Middleware)

**选型：** 看板与时间线融合视图 (`PlotLoomBoard.vue`) + `PlotAnchorMiddleware`

传统大纲只支持扁平的章节点，导致 AI 难以规划跨章节的长期伏笔。
升级后的架构引入了 `VolumeArc`（卷弧）与 `PlotAnchor`（命运锚点）。当写到锚点所在的附近章节时，`PlotAnchorMiddleware` 会扫描前瞻窗口，并向 AI 提示词中强制注入 `【命运锚点预警】` 标识，引导 AI 主动进行伏笔的回收和铺垫。

### 6. 对话式批量设定生成向导 (World Gen Wizard)

**选型：** Pinia 草稿状态流 + 关系图谱预渲染 + Tauri IPC 原子化提交

为降低重度设定本的入门门槛，系统引入 `WorldGenWizard.vue`：
1. 允许用户通过对话 Prompt 指示 AI 批量构造世界观。
2. 借助 Tool Calling (`generate_world_entities`) 输出包含 `attitude` 的角色关系集。
3. 渲染侧通过 `draftEntities` 和 `draftRelations` 状态在 `SandboxGraph.vue` 上绘制“草稿节点”（发光虚线特效）。
4. 用户审阅通过后，一键通过 `save_entity` 和 `save_state_event` 落库，实现所见即所得。

## 遗留的技术债务与妥协

### 已知风险

1. **类型定义黑洞（已临时解决）**
   - 演进期间，为了迅速联通新老接口，系统中目前存留了少量使用了 `@ts-nocheck` 的旧版文件。

2. **上下文膨胀**
   - 风险：当出场实体超过 20 人时，实体注入仍然会超过预算。
   - 现况妥协：已经加入了 `isArchived` 和 `vitalStatus: dead` 的过滤，但活人太多依然是个瓶颈，下一步需要对实体也做摘要。

3. **依赖更新断层**
   - `vis-timeline` 等第三方组件类型包在最新的 Vue3 + Vite + TS 下有些水土不服，我们临时使用了 `as any` 强制封印。

## 性能基准 (V5 实测)

### 存储估算 (SQLite 下)

| 规模 | 章节数 | 向量数 | 存储大小 |
|------|--------|--------|----------|
| 小型 | 50章 | 500-1000 | <5MB |
| 中型 | 150章 | 1500-3000 | 10-20MB |
| 大型 | 300章 | 3000-6000 | <50MB |
*(大幅优于原版设计的臃肿 JSON)*

### 响应时间 (桌面端实测)

| 操作 | 预期耗时 |
|------|----------|
| 本地 OP-RAG 检索(top-20) | ~120ms |
| 章节提取落库 | 取决于 API (通常 <5s) |
| 十九项一致性检查 | <1s |
| 万字项目冷加载 | <500ms |
