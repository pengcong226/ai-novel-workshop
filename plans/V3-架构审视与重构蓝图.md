## 三、设定系统终极方案：图元时序架构

> 融合三份研究的结论：
> - **结构化骨架**锁定不可篡改的事实（Deep Research #1 + #2 共识）
> - **自然语言血肉**承载文学性描写（叙事圣经方案的精华）
> - **时序状态切片**解决"时间静止"（DOME 时态知识图谱 + Gemini #2 的 stateTimeline）
> - **向量记忆层**召回原文细节（OP-RAG 保序检索）
> - **增量 diff + 双重验证**防止信息漂移（RFC 6902 + 提取器/验证器）

### 3.1 统一实体图元（EntityNode）

所有角色、势力、物品、地点继承自统一基元，支持跨实体的图谱关联与多跳推理。

```typescript
/**
 * 实体图元基类
 * 所有世界中的"东西"都是一个 EntityNode
 * 支持 GAM 的正交图谱存储
 */
export interface EntityNode {
  id: string                    // UUID
  type: 'CHARACTER' | 'FACTION' | 'ITEM' | 'LOCATION' | 'CONCEPT'
  name: string
  aliases: string[]             // 别名，用于去重和检索触发
  isArchived: boolean           // 逻辑销毁标记，不物理删除
  keywords: string[]            // 关键词索引
  createdAtChapter: number      // 首次出现的章节
  importance: 'critical' | 'major' | 'minor' | 'background'
}
```

### 3.2 角色模型：刚性骨架 + 时序切片 + 叙事血肉

```typescript
/**
 * 角色 = 不可变骨架 + 可变时间线 + 文学性档案
 */
export interface Character extends EntityNode {
  type: 'CHARACTER'

  // ===== 刚性骨架：代码级校验，LLM不可篡改 =====
  coreIdentity: {
    gender: 'male' | 'female' | 'other'
    birthChapter: number              // 首次出场章节
    fundamentalTraits: string         // 绝对底层设定（如"人类、无法使用魔法"）
    deathChapter?: number             // 死亡章节（如有）
  }

  // ===== 时序状态切片：每次重大变化追加一条 =====
  stateTimeline: CharacterStateSlice[]

  // ===== 关系图谱骨架 =====
  relationships: EntityRelation[]

  // ===== 叙事血肉：AI维护的自然语言档案 =====
  narrativeProfile: string            // ~300-500字，当前性格/动机/口吻
  lastNarrativeUpdate: number         // 最后更新章节号
}

/**
 * 角色在特定时间点的状态快照
 * 每当角色发生重大变化时追加一条，不覆盖旧条目
 */
export interface CharacterStateSlice {
  chapterIndex: number                // 此状态生效的章节
  vitalStatus: 'alive' | 'dead' | 'missing' | 'unknown'
  physicalState: string               // 如"健康"、"断臂"、"中毒"
  location: string                    // 当前位置
  faction: string                     // 当前阵营
  powerLevel: string                  // 当前实力等级
  currentGoal: string                 // 当前短期目标
  inventory: InventoryItem[]          // 背包
  abilities: AbilityRecord[]          // 能力清单
  trigger: string                     // 触发此状态变更的事件描述
}

export interface InventoryItem {
  itemId: string                      // 关联 EntityNode
  itemName: string
  status: 'possessed' | 'lost' | 'destroyed' | 'given_away'
  sinceChapter: number
}

export interface AbilityRecord {
  name: string
  status: 'active' | 'sealed' | 'lost'
  acquiredChapter: number
}

export interface EntityRelation {
  targetId: string
  targetName: string
  type: 'family' | 'friend' | 'enemy' | 'lover' | 'rival' |
        'master' | 'subordinate' | 'ally' | 'other'
  intensity: number                   // 0-100
  sinceChapter: number
  description: string                 // 一句话概括
}
```

### 5.2 动态 Token 预算分配

```typescript
/**
 * 根据模型窗口大小和场景类型动态分配
 */
function allocateTokenBudget(
  modelContextWindow: number,     // 如 128000
  sceneType: PlotBeat['sceneType'],
  chapterIndex: number
): TokenBudget {
  // 预留 30% 给生成输出
  const inputBudget = Math.floor(modelContextWindow * 0.7)

  // 基准比例
  let ratios = {
    system: 0.15,    // 刚性约束
    plan: 0.08,      // 大纲+节拍
    narrative: 0.12,  // 叙事档案
    rag: 0.20,       // 向量召回
    recent: 0.20,    // 最近正文
    instruction: 0.05 // 执行指令
  }

  // 场景感知调整
  if (sceneType === 'action') {
    ratios.narrative += 0.05  // 战斗需要更多能力信息
    ratios.rag -= 0.05
  } else if (sceneType === 'introspection') {
    ratios.rag += 0.08        // 回忆需要更多历史片段
    ratios.plan -= 0.03
    ratios.narrative -= 0.05
  }

  // 章节深度调整：越往后写，越需要历史召回
  if (chapterIndex > 50) {
    ratios.rag += 0.05
    ratios.recent -= 0.05
  }

  return Object.fromEntries(
    Object.entries(ratios).map(([k, v]) => [k, Math.floor(inputBudget * v)])
  ) as TokenBudget
}
```

│                      ▼                                      │
│  ┌────────────────────────────────────────────┐           │

│  │                                              │           │
│  │  4a. 骨架层更新（增量 JSON Patch）：          │           │
│  │    • 将验证通过的 stateShifts 转为            │           │
│  │      RFC 6902 JSON Patch 操作                │           │

│  │      哪些内容？请仅输出变更部分。"             │           │

## 八、总结

> **三份研究的终极共识：**
>
> 1. **不能只用结构化**（太死板，丧失文学性）——也**不能只用自然语言**（会漂移，会被篡改）→ **骨架锁事实、血肉赋灵魂**，两者缺一不可
>
> 2. **不能全量重写**（递归重写 = 对复印件再复印）→ **RFC 6902 增量 Patch + 双重验证**
>
> 3. **不能暴力堆全文**（Lost in the Middle 效应）→ **沙漏布局 + OP-RAG 保序检索**
>
> 4. **不能让时间静止**（灭门宗派仍繁华）→ **stateTimeline 时序切片 + 版本化世界实体**
>
> 5. **不能验证后直接废弃重写**（太贵）→ **局部修补反馈循环**
>
> 这套系统不再是"一个会写字的AI工具"，而是一台拥有**时序感知的实体图谱**、**动态记忆中枢**和**自我纠错能力**的小说生成引擎。
