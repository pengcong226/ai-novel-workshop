# AI小说生成项目综合评估与优化计划

## 📊 项目整体评估

**健康度评分**：7/10
**生成质量评分**：8/10
**用户体验评分**：7/10
**成本效益评分**：6/10

**一句话评价**：项目拥有极具前瞻性的三层记忆架构与表格状态机，在保障长篇小说逻辑一致性上表现出色，但底层基础架构存在新旧API割裂、长文本性能瓶颈等严重技术债务，亟待重构以释放潜力。

**核心定位**：适合需要长期创作、对世界观设定严谨度有高要求的连载小说作者与专业创作者，主要解决长篇小说生成中AI“遗忘”、“逻辑冲突”和“无意义水文”的核心痛点。

---

## 🎯 优化行动计划

### 第一阶段：核心体验保障（1-2周）
| 优先级 | 问题 | 影响 | 预计工作量 |
|--------|------|------|------------|
| P0 | 统一AI调用层，废弃旧版 `llmCaller` 的底层 `fetch` | 解决由于API层割裂导致的Rate Limit封禁、无限重试以及成本漏算问题，极大提升系统稳定性。 | 3人天 |
| P0 | 完善向量数据库后端的删除与查询方法 | 解决脏数据积累导致的检索上下文污染和幻觉问题，保证长篇生成后期的质量一致性。 | 4人天 |
| P0 | 解决大文件导出/导入的OOM风险 | 改用流式读写或分片加载，避免长篇小说序列化导致浏览器崩溃，保障核心用户体验。 | 3人天 |

### 第二阶段：效率提升（2-4周）
| 优先级 | 问题/改进点 | 影响 | 预计工作量 |
|--------|-------------|------|------------|
| P1 | 引入精准Tokenizer计算Context Token | 解决粗糙公式带来的Token估算误差，避免API 400报错和无效的上下文Token浪费。 | 2人天 |
| P1 | 重构限流器与重试策略 | 加入带随机抖动（Jitter）的指数退避和队列超时机制，解决并发时的“惊群效应”。 | 2人天 |
| P1 | 引入轻量级 LLM-as-a-Judge 评估质量 | 替代死板的正则关键词匹配，避免对使用高级修辞或复杂铺垫的优质文笔产生误判。 | 3人天 |
| P2 | 页面卸载拦截与防抖保存优化 | 增加全局 `beforeunload` 事件监听，防止用户意外关页导致最后几秒数据丢失。 | 1人天 |

### 第三阶段：能力扩展（长期）
1. **防注入与安全强化**：规范 Prompt 变量注入机制，严格区分 System 和 User 角色传递，防止大纲或角色设定触发 Prompt Injection 越狱。
2. **柔性自愈机制**：优化 JSON 自动修复策略，在截断时优先触发轻量级 LLM 纠错重试，仅将暴力闭合作为最终兜底手段。
3. **记忆扩展体系**：将 Table Memory 从简单的角色状态追踪，扩展到势力关系图谱、事件时间线联动，支持更宏大的世界观构建。

## 💎 需要保持和发扬的亮点

1. **三层递进的上下文记忆架构（Short/Mid/Long-term Memory）**
   - **核心价值**：完美解决了长篇创作中容易“遗忘”和超出 Token 限制的痛点。
   - **保持建议**：在后续迭代中，始终保留动态权重组装机制，不因单一模型Context变长而放弃分级检索。
   - **扩展建议**：支持外部设定集（如 Wiki）的自动抓取、向量化并融入长期记忆层。

2. **结构化的表格记忆与状态机机制 (Table Memory)**
   - **核心价值**：赋予 AI 类似数据库的操作权限，将隐式的文本记忆转化为严谨的状态机，使逻辑冲突发生率断崖式下降。
   - **保持建议**：作为生成管线的核心约束标准，拒绝向非结构化的纯文本隐式记忆妥协。
   - **扩展建议**：结合前端技术，将表格数据实时渲染为可视化的关系图谱或地图系统。

3. **意图拦截与 Action-Oriented AI 对话助手**
   - **核心价值**：通过将特定 JSON 意图拦截并渲染为UI操作按钮，实现了“AI推理想法-人类一键执行”的高效人机协同。
   - **保持建议**：坚持“陪你写”（被动提示辅助）而非“替你写”（强行覆盖原文）的设计哲学。
   - **扩展建议**：允许高级用户自定义拦截指令和捷径按钮，打造个性化的创作工作流。

## ⚖️ 关键权衡决策

1. **灵活性 vs 简洁性**
   - **判断**：倾向于**保留灵活性**，但在UI呈现上采取“渐进式展开”。
   - **理由**：该项目的核心竞争力在于 Table Memory 和三层记忆，目标群体多为对质量有严苛要求的专业或硬核作者。复杂的参数是刚需，但可以通过“新手模式/高级设置折叠”的设计，兼顾初学者的简洁体验。

2. **生成质量 vs 成本**
   - **判断**：坚持**质量优先，但引入严格的熔断与记账管控**。
   - **理由**：用户愿意为了逻辑一致性和好文笔承担一定 API 成本，但反感因为无限死循环重试导致的“隐形浪费”。因此保留自动重试纠错机制（欣赏者看重的亮点），但必须统一接入 `ai-service`，设置硬性重试上限（如最多3次）及每日预算熔断拦截。

3. **功能丰富 vs 用户体验**
   - **判断**：功能丰富度应服务于“不打扰的沉浸式体验”。
   - **理由**：欣赏者提出的“被动小红点建议”是神来之笔。后续所有质量检查（包括改进后的 LLM-as-a-Judge）都应作为后台静默任务，将结果温柔地挂载在侧边栏，避免像弹窗一样中断用户的创作心流。

## 💰 成本优化建议

1. **彻底堵住漏算漏洞**：立即废除直接调用 `fetch` 的旧代码，强制所有模型交互收口到带 `CostTracker` 的核心服务层，每一笔调用必须登记入账。
2. **精细化 Token 裁剪**：结合本地 Tokenizer 计算精准长度；继续发扬“触发式过滤（Trigger Send）”的优点，只将与当前场景相关的人物的状态（如最近5句话涉及的实体）发送给模型，拒绝无效的长文本堆叠。
3. **缓存复用**：针对频繁调用的轻量级判断（如质量评分、错别字检查），建立针对相同文本哈希的本地缓存，避免重复调用。

---

## 🎨 用户体验优化建议

1. **稳定高于一切**：前端最致命的体验是“写了半天结果崩溃/没存上”。必须优先解决长篇数据的增量保存与大对象导出的流式处理（Web Streams API/分块）。
2. **状态透明化**：在进行复杂的 API 重试、向量检索和后台生成时，UI 必须给出优雅的“进行中/排队中/检索中”状态反馈，消除用户的等待焦虑。
3. **可视化数据**：由于表格状态机对新手来说较为抽象，建议在UI侧增加当前章节涉及的“出场人物属性卡”，让后台的 Table Memory 能够以直观的形式被创作者检阅和修改。

## 🤔 给团队的建议

1. **技术债阻击战**：强烈建议团队在下一个迭代周期暂停所有非核心新功能的开发（Feature Freeze），开展为期一周的“技术债阻击战”。先把冗余架构（双调用层）和未实现方法（向量删除）补齐，否则大厦将倾。
2. **构建回归测试防线**：项目中拥有极其亮眼的 JSON 自动纠错和重排块（Overlap Chunking）机制，这些都是易碎品。在重构 `llmCaller` 时，必须为这些工具函数编写 Jest/Vitest 单元测试，确保核心竞争力不丢失。
3. **产品护城河**：你们的 Table Memory 是目前开源/独立开发产品中的绝对首创。不要在普通的文本生成上与大厂拼刺刀，而是应该深入挖掘“基于状态机的一致性维护”，甚至以此衍生出“剧本杀生成”、“跑团（TRPG）游戏引擎”等更广阔的应用场景。
